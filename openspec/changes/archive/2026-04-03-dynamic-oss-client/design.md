## Context

myproject-oss模块当前使用静态配置（如配置文件或硬编码）来初始化OSS客户端，这限制了系统在多环境、多租户场景下的灵活性。需要从数据库表`sys_oss_config`动态读取配置，支持阿里云OSS、MinIO等多种云存储提供商，并实现客户端的动态创建和缓存。

## Goals / Non-Goals

**Goals:**
- 从数据库动态读取OSS配置，支持运行时更新
- 支持多种云存储提供商（阿里云OSS、MinIO等）的统一接口
- 实现OSS客户端工厂，根据配置创建对应客户端
- 提供配置缓存机制，减少数据库查询频率
- 保持向后兼容，尽量减少对现有代码的破坏性修改

**Non-Goals:**
- 实现OSS操作的统一抽象层（仅关注客户端创建）
- 支持所有云存储提供商（仅限阿里云和MinIO）
- 实现配置的版本管理或审计功能
- 处理OSS操作的业务逻辑

## Decisions

### 1. 数据库访问技术
**选择**: 使用MyBatis-Plus
**理由**: 与现有技术栈保持一致，减少学习成本
**备选方案**: 
- 原生JDBC（过于底层）
- JdbcTemplate（可行但不够现代化）

### 2. 配置表结构设计
**选择**: 创建`sys_oss_config`表，包含以下核心字段：
- `id`: 主键
- `config_name`: 配置名称（唯一标识）
- `provider`: 提供商类型（aliyun, minio等）
- `endpoint`: 服务端点
- `access_key`: 访问密钥
- `secret_key`: 秘密密钥
- `bucket_name`: 存储桶名称
- `region`: 区域（可选）
- `extra_config`: JSON格式的扩展配置
- `is_active`: 是否启用
- `created_at`, `updated_at`: 时间戳

**理由**: 灵活支持多种提供商，扩展性好

### 3. 客户端工厂模式
**选择**: 使用简单工厂模式，根据`provider`字段创建对应客户端
**理由**: 简单直观，易于扩展新提供商
**实现**:
- `OssClientFactory`接口
- `AliyunOssClientFactory`、`MinioClientFactory`等具体实现
- `OssClientFactoryProvider`根据provider类型返回对应工厂

### 4. 缓存策略
使用redis

### 5. 配置刷新机制
**选择**: 提供手动刷新接口和定时刷新（可选）
**理由**: 平衡实时性和性能

## Risks / Trade-offs

### 风险
1. **数据库连接问题**: 数据库不可用时OSS功能完全失效 → 实现降级策略，使用redis缓存的最后有效配置
2. **配置格式错误**: 数据库中的配置格式错误导致客户端创建失败 → 增加配置验证和友好的错误提示
3. **性能影响**: 频繁数据库查询影响性能 → 使用缓存和批量加载
4. **安全风险**: 敏感信息（access_key等）存储在数据库中 → 考虑加密存储或使用密钥管理服务

### 权衡
1. **简单性 vs 扩展性**: 选择简单工厂模式而非更复杂的抽象工厂，便于快速实现
2. **缓存一致性 vs 性能**: 选择本地缓存而非实时查询，可能短暂使用过期配置
3. **开发速度 vs 功能完整性**: 优先支持核心提供商，后续扩展

## Migration Plan

### 部署步骤
1. 创建数据库表`sys_oss_config`并初始化测试数据
2. 部署新版本代码（保持向后兼容）
3. 逐步迁移现有配置到数据库
4. 验证功能正常后移除旧配置

### 回滚策略
1. 如果出现问题，切换回静态配置（保留旧配置文件）
2. 数据库表可保留，不影响回滚
