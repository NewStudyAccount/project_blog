## Context

项目基于 Spring Boot 3.5 + Java 21，采用多模块 Maven 结构。当前业务模块（system、blog 等）需要生成全局唯一ID，但缺少统一的分布式ID生成方案。雪花算法是业界广泛使用的轻量级方案，无需引入 Redis 或 ZooKeeper 等外部依赖，适合当前项目规模。

工具类将放置在 `myproject-framework` 模块的 `com.example.utils` 包下，与现有的框架工具类保持一致。

## Goals / Non-Goals

**Goals:**
- 提供线程安全的雪花算法ID生成工具类
- 支持自定义数据中心ID（datacenterId）和机器ID（workerId）
- 生成的ID趋势递增，兼容数据库有序索引
- 提供简洁的静态方法调用接口

**Non-Goals:**
- 不实现基于外部协调服务的 workerId 自动分配（如 ZooKeeper、Redis）
- 不实现多数据中心间的时钟同步方案
- 不提供 Spring Bean 配置方式（保持纯工具类风格）

## Decisions

### 1. 采用标准雪花算法结构（64位）

**选择**：1位符号位 + 41位时间戳 + 5位数据中心ID + 5位机器ID + 12位序列号

**理由**：这是 Twitter Snowflake 的经典结构，每毫秒可生成 4096 个ID，41位时间戳可用约69年。5+5位支持32个数据中心 × 32台机器，满足中小规模部署需求。

**备选方案**：
- 百度 UidGenerator：需要数据库支持，引入额外依赖
- 美团 Leaf：需要 ZooKeeper，架构复杂度过高
- UUID：无序，不利于数据库索引性能

### 2. 纯工具类设计（静态方法）

**选择**：使用单例模式 + 静态方法暴露，而非 Spring Bean

**理由**：工具类使用场景简单，无需依赖 Spring 容器，调用方更灵活。通过 `SnowflakeIdUtil.nextId()` 即可获取ID，降低使用门槛。

### 3. 时钟回拨处理策略

**选择**：检测到时钟回拨时抛出异常

**理由**：时钟回拨意味着可能生成重复ID，抛出异常是最安全的策略。当前项目为单机或少量实例部署，NTP 回拨概率极低。若未来需要更健壮的处理，可扩展为等待回拨恢复。

## Risks / Trade-offs

- **[时钟回拨风险]** → NTP 同步可能导致时钟回拨，当前策略为抛出异常。生产环境应确保 NTP 配置为渐进同步（不回拨）
- **[WorkerId 冲突风险]** → 多实例部署时需手动确保各实例的 workerId + datacenterId 组合唯一。当前规模下可接受，大规模部署需引入自动分配机制
- **[单机性能上限]** → 单实例每毫秒最多 4096 个ID（约400万/秒），当前业务量远未达到此上限
